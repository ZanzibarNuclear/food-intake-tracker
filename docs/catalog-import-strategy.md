# Catalog Import Strategy

The current system catalog is a bundled, deterministic JSON file generated by `npm run generate:catalog`. It seeds with `npm run db:seed:catalog`, and production upserts are keyed by lowercased food name for global system foods.

For this incremental expansion, keep the existing format:

- `tools/build_system_foods.mjs` remains the editable source list.
- `server/data/system-foods.json` remains generated output.
- Each row includes `name`, `servingDescription`, `calories`, `proteinGrams`, heuristic `nutritionScore`, heuristic `satietyScore`, and nullable `notes`.
- The database keeps `source = 'usda'` for seeded catalog rows so existing filtering/copy behavior remains unchanged.

Safe source strategy:

- Treat the bundled list as a curated common-food catalog until the schema has stable external source identifiers.
- Prefer plain common-food names and everyday serving sizes that search well.
- Avoid brand-specific rows unless a future importer stores stable source ids, brand owner, and serving metadata.
- Validate duplicate names, required fields, and non-negative numeric values before writing generated JSON.

Future bulk USDA/FDA import:

- Add a source id column or metadata table before importing thousands of rows.
- Use a pinned public USDA FoodData Central or FDA source export, stored by version/date in importer docs.
- Normalize source rows into the existing app shape, but upsert by stable source id rather than display name.
- Keep name-based aliases/search terms separate from canonical source records so renamed display labels do not create duplicate catalog foods.
